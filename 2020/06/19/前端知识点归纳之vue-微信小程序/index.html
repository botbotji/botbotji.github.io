v-show 和 v-if 的区别及应用场景

  • 相同点：v-show 和 v-if 都能对元素控制其显示和隐藏；
  • 不同点：
    1. v-if 是通过控制 DOM 节点的存在与否来控制元素的显隐;
      v-show 是通过设置 Dom 节点元素的 display 样式，block 为显示，none 为隐藏;
    2. 编译过程：v-if 切换有一个局部编译/卸载的过程，切换过程中合适的销毁和重建内部的事件监听和子组件;v-show 只是简单的基于 css 的切换;
    3. 编译条件:
      • v-if 是惰性的，如果初始化条件为 false，则什么都不会做；只有在条件第一次为 true 时才开始局部编译(编译被缓存？编译被缓存后，然后再切换的时候进行局部卸载);
      • v-show 是在任何条件下(首次条件是否为真)都会被编译，然后被缓存，而且 DOM 元素保留
    4. 性能消耗：v-if 有更高的切换消耗；v-show 有更高的初始渲染消耗；
    5. v-if 有配套的 v-else-if 和 v-else，而 v-show 没有
    6. v-if 可以搭配 template 使用，而 v-show 不行
  • 应用场景：
    • 非常频繁的切换 则使用 v-show 较好(v-if 会添加和删除 DOM 节点);
    • 如果在运行时条件很少改变，则使用 v-if 较好。
  • 小结：
    • v-if 判断是否加载，可以减轻服务器的压力，在需要时加载,但有更高的切换开销，比如单页面应用;
    • v-show 调整 DOM 元素的 CSS 的 dispaly 属性，可以使客户端操作更加流畅，但有更高的初始渲染开销。如果需要非常频繁地切换，则使用 v-show 较好，tab 中使用较好；

订阅消息

  • 模板消息官网：https://developers.weixin.qq.com/miniprogram/dev/framework/open-ability/template-message.html
  • 2020 年 1 月份微信去掉了模板消息，换成了模板消息
  • 之前模板消息，是通过 formId 通过后台发出模板消息的
  • 页面的 form 组件，属性 report-submit 为 true 时，可以声明为需要发送模板消息，此时点击按钮提交表单可以获取 formId，用于发送模板消息。或者当用户完成 支付行为，可以获取 prepay_id 用于发送模板消息。
  • 这里要取消 report-submit 这个属性,否则会有报错

维护旧项目注意点

  • 维护旧项目，在添加需求的时候，要从首页起阅读源码，判断其执行逻辑与流程，若直接进入添加需求的页面。可能导致项目需求没得完成；

微信小程序之内嵌网页(webview)

  • 官方介绍:

    • 承载网页的容器。会自动铺满整个小程序页面，个人类型的小程序不支持使用。

    • 微信的基础库自 1.6.4 开始，就内嵌了一个组件来连接我们的 HTML 页面

    • 具体开发流程：

      1. 登录小程序平台管理后台页面，进入“设置” => “开发设置”，配置业务域名(注意：你需要在弹出的窗口里下载“校验文件”，并把这个校验文件上传到你的网站的根目录，供小程序平台进行验证，验证通过了才能成功添加域名。)
      2. 总是自动铺满整个页面，且每个页面只能有一个，它会覆盖其他组件。也就是说，没有办法实现小程序界面组件和页面混排的情况，这点要注意。
      3. 页面跳转:利用微信的 JSSDK，使用它提供的方法来实现相关跳转功能
