Java消息中间件+ActiveMQ(七)消息存储和持久化

发表于 2020-02-29 |
字数统计: 3.5k | 阅读时长 ≈ 15

消息存储

  • MQ的高可用：事务，持久，签收，可持久化
  • 事务，持久，签收是MQ自带的，而可持久化是未界
  • 可持久化：
    1. Replicated LevelDB Store (带复制功能的LevelDB)；
    2. KahaDB (可汗DB)
    • 说明：
      1. 可持久化就是数据高可用的保障手段，若MQ服务器挂了，消息不会丢失的机制。
      2. 为了避免意外司机以后丢失信息，需要做到重启后可以恢复消息队列，消息系统一般都会采用持久化机制。
      3. ActiveMQ的消息持久化机制有JDBC，AMO，KahaDB和LevelDB，无论使用哪种持久化方式，消息的存储逻辑都是保持一致的。
      4. 就是在发送者将消息发送出去后，消息中心首先将消息存储到本地数据文件，内存数据库或者远程数据库等在试图将消息发送给接收者，成功则将消息从存储中删除，失败则继续尝试发送。
      5. 消息中心启动以后首先要检查指定的存储位置，如果有未发送成功的消息，则需要把消息发送出去。

        KahaDB消息存储(默认 文件系统存储在本地硬盘)

  • 基于日志，从ActiveMQ5.4开始默认使用的持久化插件
  • 说明：
    1. KahaDB是目前默认的存储方式，可用于任何场景，提高了性能和恢复能力
    2. 消息存储用一个事务日志和仅仅用一个索引文件来存储他所有的地址。(比如，字典 的目录就是索引文件 正文就是事务日志)
    3. KahaDB是一个专门针对消息持久化的解决方案，它对典型的消息使用模式进行了优化。
    4. 数据被追加到data logs中。当不再需要log文件中的数据时，log文件会被丢弃(删除/归档)。

  • 存储原理
    1. kahaDB在消息保存目录中只有4类文件和一个lock，跟ActiveMQ的其他几种文件存储引擎相比这就非常简介了。
    2. db-.log kahaDB存储消息到预定义大小(32兆)的数据记录文件中，文件命名为db-.log。当数据已满是，一个新的文件随之创建，number数值也会随之递增，它随着消息数量的增多，如每32M一个文件，文件名按照数字进行编号，如db-1.log，db-2.log……。当不在有引用到数据文件中的任何消息时，文件被删除或归档。
    3. db.data该文件包含了支持的BTree索引,索引了消息记录中的消息，它是消息的索引文件，本质是B-Tree(B树)，使用B-Tree作为索引指向db-.log里面存储的消息。
    4. db.free 当前db.data文件里那个是空闲的，文件具体内容是所有空闲页的ID。
    5. db.redo 用于进行消息恢复，如果KahaDB消息存储在强制退出后启动，用于回复BTree索引。(相当于备份2和3)
    6. lock 文件锁，表示但前获取kahadb读取写权限的broker。

      LevelDB消息存储(了解)

  • 这种文件系统是从ActiveMQ5.8之后引进的，它和kahaDB非常相似，也是基于文件的本地数据库存储的形式，但是它提供比kahaDB更快的持久性。
  • 但它不使用自定义BTree实现来索引预写日志，而是使用基于LevelDB的索引

  • 4.建仓SQL和建表说明：
    • 建一个名为activemq的数据库 create database activemq
    • ActiveMQ重启会自动创建三张表
      • ACTIVEMQ_MSGS(消息)
        • 表中的字段
          ID：自增数据库主键
          CONTAINER:消息的Destination(目的地即队列/主题)
          MSGID_PROD:消息发送者的主键
          MSG_SEQ:是发送消息的顺序，MSGID_PROD+MSG_SEQ可以组成JMS的MessageID
          EXPIRATION:消息的过期时间，存储的是1970-01-01到现在的毫秒数
          MSG:消息本体的Java序列化对象的二进制数据
          PRIORITY:优先级，从0-9，数值越大优先级越高
      • ACTIVEMQ_ACKS(签收)
        • 表中的字段
          • activemq_acks:用于存储订阅关系。如果是持久化Topic，订阅者和服务器的订阅关系在这个表中保存。
          • 数据库的字段如下：
            CONTAINER:消息的Destination(目的地即队列/主题)
            SUB_DEST:如果是使用Static集群，这个字段会有集群其他系统消息
            CLIENT_ID:每个订阅者都必须有一个唯一的客户端ID用于区分
            SUB_NAME:订阅者名称
            SELECTOR:选择器，可以选择只消费满足条件的消息。条件可以用自定义属性实现，可支持多属性AND和OR操作
            LAST_ACKED_ID：记录消费过的消息的ID
        • 表存储持久化订阅的信息和最后一个持久订阅接收消息ID
      • ACTIVEMQ_LOCK(锁定)
        • 在集群环境中才能有用，只有一个Broker可以获得消息，称为Master Broker(主Broker)，其他的只能作为备份等待Master Broker不可用，才可能成为下一个Master Broker。这个表用于记录那个Broker是当前的MaSter Broker

    代码运行验证：

  • 一定要开启持久化：messageProducer.setDeliveryMode(DeliveryMode.PERSISTENT);
  • 在点对点的类型中:
    • 当DeliveryMode设置为NON_PERSISTENCE(非持久化)时，消息被保存在内存中；
    • 当DeliveryMode设置为PERSISTENCE(持久化)时，消息被保存在broker的相应的文件或者数据库中；
    • 而且点对点类型中消息一旦被Consumer就从broker中删除(broker就是ActiveMQ的一个实例)
  • 在主题订阅/发布类型中：
    • 数据不会被删除，

      小结：

  • 如果是队列Queue：
    • 在没有消费者消费的情况下会将消息保存到activemq_msgs表中，只要有任意一个消费者已经消费过了，消费之后这些消息将会立刻被删除
  • 如果是主题Topic：
    • 一般是先启动消费订阅然后在生产的情况下会将消息保存到activemq_acks里；

JDBC Message store with ActiveMQ Journal （JDBC消息存储增强带高速缓存日志）

  • 这种方式克服了JDBC Store的不足，JDBC每次消息过来，都需要去写库和读库。
  • ActiveMQ Journal，使用高速缓存技术，大大提高了性能。
  • 当消费者的消费速度能够及时跟上生产者消息的生产速度时，journal文件能够大大减少需要写入的DB中的消息。
  • 举例子说明：
    • 生产者生产了1000条消息，这1000条消息会保存到journal文件，如果消费者的消费速度很快的情况下，在journal文件还没有同步到DB之前，消费者已经消费了90%的以上的消息，那么这个时候只需要同步剩余的10%的消息到DB。
    • 如果消费者的消费速度很慢，这个时候journal文件可以使消息以批量方式写到DB。

      总结：

    • 持久化消息主要指：
      MQ所在的服务down(挂掉)了消息不会丢失的机制
    • ActiveMQ的消息持久化机制有：
      AMQ：基于日志文件
      KahaDB：基于日志文件，从ActiveMQ5.4开始默认的持久化插件
      JDBC：基于第三方数据库
      LevelDB:基于文件的数据库存储，从ActiveMQ5.8之后推出的LevelDB的持久化引擎性能高于KahaDB
      Replicated LevelDB Store ：从ActiveMQ5.9提供了LevelDB和Zookeeper的数据复制方式，用于Master-slave方式的首选数据复制方案。
    • 无论哪种持久化机制，消息的存储逻辑都一致的。
